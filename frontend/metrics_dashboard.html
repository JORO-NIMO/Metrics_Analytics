How We Collect Data
First Degree
👥 Direct to Participants
Access to the people themselves.
Methods: Interviews, questionnaires, think-aloud, work diaries
In our project: The survey form (submitsurvey.php) collects first-degree data directly from mothers using the app.
Survey questions ask about tracker usefulness, tip accuracy, and ease of use.
Endpoint: submitsurvey.php
Second Degree
🖥️ Work Environment (Real-time)
Access to environment while work happens — not people directly.
Methods: Instrumented systems, runtime monitoring, event logging
In our project: MetricsLogger auto-instruments every PHP action — tracker uses, logins, page views, errors — all logged in real-time without user involvement.
Class: MetricsLogger (all logXxx methods)
Third Degree
📁 Work Artifacts Only
No access to people or live environment — only artefacts (logs, code, documents).
Methods: Bug report analysis, offline log mining, code review
In our project: Analysis of tracker_logs, system_errors, content_feedback tables after-the-fact; SQL view queries on stored data.
Tables: tracker_logs, system_errors, content_feedback
Research Quality Standards — Compliance Checklist

6-area checklist ensuring our research meets international quality standards

Area 1
📌 Experimental Context
✅ C1: Context fully specified — entities (users, tracker, tips), attributes (error rate, coverage %), measures (ratio scale)
✅ C2: Hypothesis stated BEFORE data collection: "Validation reduces errors by >50%"
✅ C3: Exploratory questions stated upfront in experiment registry (experiments table)
Area 2
🎯 Experimental Design
✅ D1: Population: mothers using MHU app in Uganda
✅ D3: Assignment: control (no validation) vs treatment (with validation)

Core Testing Principles
Principle 1
🔁 Replication

Same conditions should produce same results. Each tracker calculation is independently logged — multiple users performing the same action create replicated observations. This eliminates confounded results from one-off measurements.

Implemented via tracker_logs (all uses recorded)
Principle 2
🎲 Randomization

Random assignment minimises uncontrolled variable effects. In our experiment, users are randomly assigned to control/treatment groups based on session ID hashing — no bias in group assignment. This reduces confounding from user skill level.

Field: group_name in experiment_observations
Principle 3
🧩 Local Control (Blocking)

Blocking groups subjects by a known confounding factor (e.g. mobile vs desktop users). Balancing keeps equal numbers per group. Correlation analysis uses experiment_results mean/std data to find relationships between variables.

Fields: block_name in experiment_observations
